# 次数与积分运营说明

本文说明当前小程序的免费次数、积分、兑换码和日常运维方式。系统目前**未接入微信支付**，代码中的套餐仅用于展示和后续支付接入，不会产生真实交易。

## 1. 计费规则

每个用户拥有“当日免费次数”和“积分余额”：

1. 新用户首次建立积分账户时，赠送 `FREE_SIGNUP_CREDITS` 积分。
2. 每天优先使用免费次数；免费次数用完后，每次生成消耗 `CREDITS_PER_GENERATION` 积分。
3. 积分生成任务提交时先预扣，生成成功后确认消费，生成失败或服务中断时自动返还。
4. 失败的免费任务不占用免费次数。
5. “今日剩余次数”是免费次数与当前积分可生成次数之和。

当前默认值：

| 环境变量 | 默认值 | 说明 |
| --- | ---: | --- |
| `DAILY_FREE_LIMIT` | `10` | 每位用户每天的免费生成次数 |
| `FREE_SIGNUP_CREDITS` | `3` | 新用户首次赠送积分 |
| `CREDITS_PER_GENERATION` | `1` | 每次生成消耗积分 |
| `ADMIN_TOKEN` | 未配置 | 管理接口令牌；未配置时关闭管理接口 |

配置写在 `server/.env`，修改后需要重启后端。以上数值应使用正整数。当前“每日”按 SQLite 的 UTC 日期统计，即北京时间每天 08:00 重置。

## 2. 查找用户账号

让用户打开小程序“我的”页，顶部会显示：

```text
账号 #123
```

其中 `123` 就是后续查询或充值使用的 `userId`。操作前应让用户再次确认账号，避免给错用户充值。

## 3. 使用命令行管理

以下命令均从仓库根目录执行。命令直接操作 `server/data/app.db`，不需要 `ADMIN_TOKEN`，因此只能让可信任的服务器管理员使用。

### 查询余额

```bash
node server/scripts/credits.js balance 123
```

### 增加或扣减积分

```bash
# 给账号 #123 增加 20 积分
node server/scripts/credits.js grant 123 20 客服补偿

# 从账号 #123 扣减 5 积分
node server/scripts/credits.js grant 123 -5 误充撤销
```

每次调整都会写入积分流水。扣减前应先查询余额，不要把余额调整为负数。

### 生成兑换码

```bash
# 生成 10 个兑换码，每个兑换码增加 20 积分，每个码限用 1 次
node server/scripts/credits.js gen-code 20 10

# 生成 1 个可被 5 位不同用户使用的兑换码
node server/scripts/credits.js gen-code 20 1 5
```

单次最多生成 500 个兑换码。即使兑换码允许多人使用，同一用户也只能使用该兑换码一次。兑换码应按需生成并通过私聊发给用户，不要公开发布未设总使用次数的兑换码。

### 查看最近生成的兑换码

```bash
node server/scripts/credits.js list-codes
```

命令会显示最近 100 个兑换码、积分面值、已用次数、最大次数和状态。

## 4. 使用管理员 HTTP 接口

只有在 `server/.env` 配置了随机且足够长的 `ADMIN_TOKEN` 后，管理员接口才会开启。令牌必须放在请求头 `x-admin-token` 中，不能放在 URL、前端代码或仓库中。

### 给用户调整积分

```bash
curl -X POST https://api.example.com/api/admin/credits/grant \
  -H "Content-Type: application/json" \
  -H "x-admin-token: $ADMIN_TOKEN" \
  -d '{"userId":123,"amount":20,"reason":"客服补偿"}'
```

### 批量生成兑换码

```bash
curl -X POST https://api.example.com/api/admin/redeem-codes \
  -H "Content-Type: application/json" \
  -H "x-admin-token: $ADMIN_TOKEN" \
  -d '{"credits":20,"count":10,"maxUses":1,"reason":"活动发放"}'
```

HTTP 接口单次最多生成 200 个兑换码。日常服务器操作优先使用 CLI；需要接运营后台时再使用 HTTP 接口。

## 5. 用户兑换流程

用户进入小程序“我的”页，点击“兑换码充值”，输入兑换码即可。系统会检查：

- 兑换码是否存在、有效或过期；
- 总使用次数是否已经耗尽；
- 当前用户是否已经使用过该兑换码。

兑换成功后积分立即到账，并显示在“我的”页和首页额度提示中。

## 6. 调整运营参数

编辑生产环境的 `server/.env`：

```dotenv
DAILY_FREE_LIMIT=3
FREE_SIGNUP_CREDITS=3
CREDITS_PER_GENERATION=1
ADMIN_TOKEN=请替换为随机长字符串
```

使用 PM2 部署时加载新配置：

```bash
pm2 restart ai-outfit --update-env
curl https://api.example.com/health
pm2 logs ai-outfit --lines 100
```

减少免费次数只影响之后的额度计算，不会清空现有积分。修改注册赠送积分只影响尚未建立积分账户的新用户。

## 7. 成本与定价

生图模型价格会随模型、分辨率和参考图数量变化，应以火山方舟控制台的最新账单和价格页为准。上线收费前至少统计：

- 每张成功输出图的模型成本；
- 多张参考图产生的输入成本；
- 衣物分类视觉模型、服务器、存储和带宽成本；
- 微信支付手续费、退款和活动赠送损耗；
- 目标毛利率。

可用以下公式估算套餐最低售价：

```text
套餐最低售价 =
  套餐生成次数 × 单次综合成本
  ÷ (1 - 目标毛利率 - 支付费率)
```

`server/index.js` 中当前展示的体验包、标准包和超值包只是占位配置，且接口明确返回 `purchaseEnabled: false`。接入真实支付前必须根据最新成本重新定价，不能直接把占位价格用于销售。

## 8. 发布与验证

后端更新的一般流程：

```bash
git pull origin main
cd server
npm ci --omit=dev
pm2 restart ai-outfit --update-env
curl https://api.example.com/health
```

发布后至少验证：

1. 新用户登录后收到正确的注册赠送积分；
2. 免费生成成功后，免费剩余次数减少且积分不变；
3. 免费额度用完后，成功生成会扣除积分；
4. 生成失败后积分已返还；
5. 兑换码只能按设定次数使用；
6. 免费次数和积分都不足时，生成接口会拒绝请求；
7. 服务重启后，中断任务被标记失败且预扣积分已返还。

小程序上传前还应确认：

- `miniprogram/config.js` 指向正确的生产 API；
- 微信后台的 `request` 和 `downloadFile` 合法域名已配置；
- 上传私钥只保存在受控上传机，权限设为仅所有者可读，绝不提交到 Git；
- 先发布体验版完成真机回归，再提交正式审核。

## 9. 数据备份与安全

核心数据位于：

- `server/data/app.db`：账号、积分、兑换码、衣柜和历史记录；
- `server/data/uploads/`：用户上传图片和生成结果。

备份必须同时覆盖数据库和图片目录。运行中的 SQLite 数据库不要只复制单个 `app.db` 文件；应使用 SQLite 在线备份命令，或短暂停止后端后完整备份 `server/data/`。

以下内容不得提交到仓库或发送到公开聊天：

- `server/.env`；
- `WX_SECRET`、`ARK_API_KEY`、`VISION_API_KEY`；
- `ADMIN_TOKEN`；
- 微信代码上传私钥；
- 生产数据库和用户图片。

如任何密钥出现在日志、聊天或提交历史中，应立即在对应平台轮换，不要继续用于正式环境。
